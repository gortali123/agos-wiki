## [2026-07-10] develop | bug fix: PROGRESSIVO_CONTROPARTE sempre NULL in variazioni_anagrafiche

L'utente ha segnalato che `PROGRESSIVO_CONTROPARTE` in `variazioni_anagrafiche.sql` è sempre `NULL`. Causa: nel calcolo finale (`COALESCE(H.OLD_PROGRESSIVO, MAX(H.OLD_PROGRESSIVO) OVER (PARTITION BY CD_CONTROPARTE) + ROW_NUMBER() OVER (...))`), quando una `CD_CONTROPARTE` non ha nessuna riga esistente a target (controparte nuova, o il primo run/full-refresh dove il target è vuoto), tutte le righe del gruppo hanno `OLD_PROGRESSIVO = NULL` (valore hardcoded per le righe "nuove" del ramo 1) — quindi `MAX(...)` su quella partizione è `NULL`, e `NULL + ROW_NUMBER()` resta `NULL`. Confrontando con `variazioni_anagrafiche.sql.old` (versione pre-ottimizzazione incrementale), lì il `MAX(...)` equivalente era esplicitamente `COALESCE(MX.MAX_PROGRESSIVO_CONTROPARTE, 0)` — il fallback a 0 è andato perso nella riscrittura del pattern "delta + last-open-row" ([[ottimizzazione-variazioni-anagrafiche-scd2]]). Il bug si autoalimenta: al primo run tutte le controparti sono "nuove" quindi ogni riga a target parte già con `PROGRESSIVO_CONTROPARTE = NULL`; ai run successivi anche le controparti "esistenti" ereditano quel `NULL` da `T.PROGRESSIVO_CONTROPARTE` (ramo 2), perpetuandolo per sempre — spiega il "sempre NULL" osservato, non solo al primo run. Fix applicato in `develop/models/L2/ANAGR_CONTROPARTE/variazioni_anagrafiche.sql`: reintrodotto `COALESCE(MAX(H.OLD_PROGRESSIVO) OVER (PARTITION BY H.CD_CONTROPARTE), 0)`. Non testato su dati reali. Nota: se il target reale ha già righe con `PROGRESSIVO_CONTROPARTE = NULL` da run precedenti, serve un `--full-refresh` (o backfill mirato) dopo aver applicato il fix — i valori NULL già persistiti non si autocorreggono con un run incrementale.
